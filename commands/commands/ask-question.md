# /ask-question $ARGUMENTS.file

## 목적

주어진 MD 파일의 내용을 기반으로 심층 인터뷰를 진행하여 스펙을 완성한다.

## 실행 단계

### 1. 파일 읽기

`$ARGUMENTS.file` 파일을 읽고 내용을 파악한다.

### 2. 심층 인터뷰 진행

AskUserQuestion 도구를 사용하여 다음 측면들에 대해 심층적으로 질문한다:

**질문 영역:**
- 기술적 구현 (아키텍처, 기술 스택, 성능 요구사항)
- UI/UX (사용자 흐름, 인터랙션, 접근성)
- 비즈니스 요구사항 (목표, 성공 지표, 우선순위)
- 엣지 케이스 (예외 상황, 에러 처리)
- 트레이드오프 (시간 vs 품질, 확장성 vs 단순성)
- 제약 사항 (기술적 한계, 리소스, 일정)
- 의존성 (외부 서비스, API, 데이터)
- 보안 및 우려 사항

**질문 원칙:**
- 뻔하거나 상투적인 질문 금지
- "왜?"를 파고드는 심층 질문
- 구체적인 시나리오 기반 질문
- 모호한 부분을 명확히 하는 질문
- 숨겨진 가정을 드러내는 질문

**예시 (하지 말아야 할 질문):**
- "사용자는 누구인가요?" (너무 일반적)
- "어떤 기능이 필요한가요?" (이미 파일에 있을 것)

**예시 (해야 할 질문):**
- "동시에 100명이 접속했을 때 응답 시간 목표는?"
- "네트워크 끊김 상태에서 사용자 입력은 어떻게 처리?"
- "이 기능과 기존 X 기능 간의 우선순위 충돌 시 어떻게?"
- "MVP에서 반드시 빠져야 할 기능은?"

### 3. 반복 인터뷰

- 각 질문에 대한 답변을 받으면, 후속 질문으로 더 깊이 파고든다
- 모든 영역이 충분히 명확해질 때까지 계속한다
- 답변이 모호하면 구체적인 예시를 요청한다
- 인터뷰 중 발견한 새로운 의문점도 즉시 질문한다

### 4. 인터뷰 종료 조건

다음 조건을 모두 만족하면 인터뷰를 종료한다:
- 모든 주요 영역에 대해 구체적인 답변을 얻음
- 트레이드오프와 우선순위가 명확함
- 엣지 케이스 처리 방안이 정의됨
- 기술적 구현 방향이 결정됨

### 5. 스펙 문서 작성

인터뷰 내용을 바탕으로 원본 파일에 스펙을 추가하거나, 별도 스펙 파일을 생성한다.

**스펙 문서 구조:**
```markdown
## 스펙 (인터뷰 기반)

### 요약
- 핵심 목표
- 주요 결정 사항

### 기술 스펙
- 아키텍처
- 기술 스택
- 성능 요구사항

### UI/UX 스펙
- 사용자 흐름
- 주요 인터랙션
- 반응형 고려사항

### 엣지 케이스
- 시나리오별 처리 방안

### 제약 및 트레이드오프
- 선택한 방향과 근거

### 미결정 사항
- 추후 결정 필요한 항목
```

### 6. 사용자 확인

작성된 스펙을 보여주고 수정이 필요한지 확인한다.

## 출력

인터뷰 완료 후:
- 원본 파일에 스펙 추가 또는
- `[원본파일명]-spec.md` 형식으로 별도 파일 생성
