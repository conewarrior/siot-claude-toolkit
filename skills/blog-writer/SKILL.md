# Blog Writer Skill

siot 포트폴리오 블로그에 올릴 글을 작성하는 스킬입니다.

## 블로그 폴더 구조

각 블로그 글은 **폴더** 단위로 관리합니다:

```
docs/content/blog/
├── github-api-claude-commands/
│   ├── index.mdx          # 본문
│   └── images/            # 이미지 폴더
│       ├── before.png
│       └── after.png
└── ...
```

**폴더명**: 영어 케밥케이스 (예: `my-new-post`)
**본문 파일**: `index.mdx`
**이미지 폴더**: `images/` (필요시 생성)

## Frontmatter

```yaml
---
title: "제목"
description: "한 줄 설명 (리스트에 표시됨)"
date: "YYYY-MM-DD"
category: "개발"  # "디자인" | "개발"
published: true
---
```

## 이미지 사용법

이미지는 해당 글의 `images/` 폴더에 저장하고, **상대 경로**로 참조:

```markdown
![설명](./images/screenshot.png)
```

커밋 시 pre-commit hook이 자동으로:
1. 로컬 이미지를 GitHub에 업로드
2. 경로를 jsDelivr CDN URL로 변환

## 작성 톤 & 무드

### 핵심 원칙

- **평서체** 사용 ("~했습니다" X, "~했어요" X, "~했다" O)
- **탐구적 접근**: "왜 그런가?"라는 질문에서 시작
- **당연함에 대한 의문**: 오랫동안 정형화된 규칙의 근거를 파헤침
- **겸손한 자신감**: 한계도 솔직히 인정하면서 인사이트 공유

### 글의 성격

- 진지하고 학구적이되 지루하지 않게
- 전문적이지만 접근 가능한 언어 사용
- 실험적이고 도전적인 태도
- 실무에서 바로 적용 가능한 내용 지향

### 피해야 할 것

- 뻔한 일반론으로 시작하기
- 근거 없는 주장
- 과도한 수식어나 감탄사
- "쉽다", "간단하다" 같은 표현 남발

## 글 구조

### 스타일 A: 탐구/연구형 (디자인 시스템, 이론 분석)

당연하게 여겨지던 것의 "왜?"를 파헤치는 글.

**1. 문제 제기 / 의문**
- 당연하게 받아들여지던 관행에 질문 던지기
- "왜 그런 규칙이 생겼을까?"
- 예: "컬러 스케일을 세 자리 숫자로 쓰는 이유가 뭘까?"

**2. 배경 지식**
- 이해에 필요한 개념 설명
- 관련 이론이나 표준 소개 (WCAG, CIE 등)
- 구체적인 수치와 공식 제시

**3. 분석 / 실험**
- 실제 사례 분석 (Adobe, Google 등)
- 직접 실험한 과정과 결과
- 이미지와 도표로 시각적 설명

**4. 결론 / 한계**
- 발견한 인사이트 정리
- 한계점 솔직히 인정
- 추후 연구 방향 제시 (선택)

### 스타일 B: 튜토리얼/경험형 (개발 워크플로우)

문제를 해결한 과정을 공유하는 글.

**1. 도입부 (Hook)**
- 문제 상황 또는 불편함을 간결하게 서술
- "~하다 보니 ~가 불편했다"

**2. 기존 방식 (Before)**
- 처음에 어떻게 해결했는지
- 어떤 한계가 있었는지

**3. 개선 과정 (Journey)**
- 어떤 계기로 개선하게 되었는지
- 어떤 대안들을 고려했는지

**4. 새로운 방식 (After)**
- 개선된 방식 설명
- 핵심 코드나 설정

**5. 마무리**
- 배운 점 또는 인사이트

## 좋은 글의 특징

### 구체성
```
❌ "명도가 중요하다"
✅ "WCAG에서는 일반 텍스트에 최소 4.5:1 이상의 명암비를 요구한다"
```

### 실제 사례 활용
```
❌ "대기업들은 이런 방식을 쓴다"
✅ "Adobe Design Spectrum의 그린 스케일을 분석해보면..."
```

### 과정의 투명성
```
❌ "결국 이 방법이 최선이다"
✅ "두 개의 변수로 일관성 있는 컬러 스케일을 만들기에는 그 범위가 너무 넓었다. 변수를 하나로 줄일 필요가 있었다."
```

### 한계 인정
```
❌ "이 방법으로 완벽하게 해결된다"
✅ "일정 범위까지는 일관된 컬러 스케일을 구현할 수 있었으나, CIElab 모델만으로는 한계가 있었다."
```

## 시리즈 글 작성 시

연속되는 주제의 글을 쓸 때:

- 이전 글 내용을 간단히 요약하며 시작
- "앞서 우리는 ~를 알아보았다"
- 이번 글의 목적을 명확히 제시
- 다음 글 예고로 마무리 (선택)

## 새 글 작성 체크리스트

1. [ ] `docs/content/blog/[slug]/` 폴더 생성
2. [ ] `index.mdx` 작성
3. [ ] 이미지가 있으면 `images/` 폴더에 저장
4. [ ] 이미지 경로는 `./images/파일명.png` 형식
5. [ ] frontmatter의 date를 오늘 날짜로 (YYYY-MM-DD)
6. [ ] 평서체로 작성했는지 확인
7. [ ] 구체적인 수치/사례가 포함되었는지 확인
